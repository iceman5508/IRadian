Download

IRadian is a lightweight, component-based, PHP framework. The framework was designed and modeled after the Angular 2/4 framework. So, for users of Angular 2/4 the structure and use of the framework will be slightly similar. However, If you have no Angular experience, the framework is very easy to learn. This is especially true since users can run native php directly in the template (not recommended, but if need be.). Users will be able to create safe, fast, robust, modular driven applications. So let us begin and learn IRadian through our demo application.

If you have not done so yet, download the framework and follow the installation instructions below.

Download IRadian

Demo Project: My Video Site

  • Use built-in UI loaders to load UI libraries and application data.
  • How to build and use components
  • How to build and use api with the framework
  • Using built in routing to navigate to different views
  • Control data with IRadian template based if and loops
  • Using built-in module loaders to load in modules
  • Interact with a database
  • Have a deep understanding of iComponents and their importance to the application

loaders
#config[title],#ui[bootstrap_css],#config[title] and #ui[jquery_js, bootstrap_js] are all examples of loaders.

Loaders are ways to include ui libraries, project data, and modules into the template. IRadian currently has three different types of loaders.

The first of these loaders is the #config loader. This loader bring application information to the template. Currently in our app.html file, we see that the loader is being used to bring in the 'title' variable as in #config[title]. config loaders are the only loaders in which users can set the values to. This can be done in the iConfig file. Additionally, users are not restricted to the default variables in that file as any variable added to the project array can be accessed as a loader.

The second of the loaders is the #ui loader. This loader brings UI libraries to the application and template. Currently in the app.html file, we see that the loader is being used to bring in the bootstrap css library as in #ui[bootstrap_css] and the jquery and bootstrap javascript libraries with #ui[jquery_js, bootstrap_js]. ui are external libraries that IRadian developers believed are commonly used my many developers. Full list of ui loader commands and their libraries are as follows:

  • bootstrap_css = bootstrap css file
  • bootstrap_js = bootstrap js file
  • jquery_css = jquery css file
  • jquery_js = jquery js file
  • imodal_css = modal css file
  • imodal_js = modal js file

The final type of loader is the #module loader. This loader brings in javascript libraries written by IRadian developers to the application and template. At the moment in our code we are not using this loader. But no worries, we will use it later in the project. Calling this loader is the same as the previous two, simply call the #module[whatever library/libraries you want to load]. Full list of current module loader commands and their purpose are as follows:

  • iEnvironment = This module hold information about the web environment such as browser size, and mobile device type.
  • iHttpService = I believe the name of this module says it all. Note on this module, it requires jquery. So uses the #ui[jquery_js] loader with it.
  • iRedirect = This module handles page redirects.
  • iVariable = This module handles variable validation
  • iWeb = This module is the js equivalent of the iWeb class

One final important note about loaders, is that they can only be loaded in the application (main) template file. Loaders will not work if loaded in component template files.
Components and component template files
Along with the loaders, another strange code that was in the app.html file is {#content}. Loaders and this {#content} call are all apart of the iTags sub-language that is built into the framework. Users will never use the language directly, rather it is a language the core parts of the framework use to communicate with each other. As for the {#content}, the application template (app.html) is telling the framework to bring in the component and view with the signature of content.

Now let us take a closer look as to what components are, how they are created, registered with the application, and called into view.

What are components

Components are self contained html code that relate to a specific view. In short, components are modular views. For example, in our current application, we have one component. That component is the content component, and it is responsible for all that we see with the exception of the navigation bar and footer.

How are components created.

To understand how components are created, let us create a component together. In the app folder, create a new folder called header. Then in that folder create two files:

1) header.php
2) header.html

Both file names do not have to be the same, but as good coding practice it is recommended that both files do have the same name.

The header.php file is the component/view model that will handel all the data aspect of our view. each component and only alter the view that they are connected to. In this respect, the data scope is local only to that specific view.

The header.html file is the component template/view. This file contains all the html, and display data. It is important to note, that each component share style detail from whatever css is presented in the application template. For example, since bootstrap is loaded in our app.html/application template, each component can use bootstrap.
